From quieting to learning

Turn the agent's repeated expensive work into a reusable, deterministic asset.

quiet-bash already shrinks each result the moment it's produced. The next step is to make it learn — notice where tokens really go, recognize what recurs, and compile that work into something cheap that runs deterministically next time.

01 · The shift

Cache the procedure, not just the answer.

Re-serving a stored result is the simplest move. The bigger lever is compiling a repeated LLM-driven loop into a deterministic artifact — a script, a hook, a markdown skill, an MCP tool — so future runs skip the model entirely. The same recognize-and-reuse instinct already lives in quiet-dedup.sh; this widens it on every axis.

02 · The loop

Six stages, narrow interfaces.

Each stage is independently useful and independently shippable. Stage 1 alone measures where tokens go; stages 1–2 produce a "what's worth crystallizing" report; stage 3 onward is where tokens are actually saved.

01

Observe

One ledger row per tool call: fingerprint, tokens, cost, inputs. Append-only — zero prompt-cache impact.

02

Recognize

Fingerprint + frequency-threshold mining, ranked by recurrence × cost.

03

Crystallize

Pick the lowest rung that covers the pattern; synthesize mechanically, by LLM, or via review.

04

Verify

Differential-test vs the golden result; sandbox; sign & pin before promotion.

05

Reuse

Retrieve (exact key / vector-db / progressive disclosure) and serve cache-safe at the tail.

06

Maintain

Shadow re-check, invalidate on drift, retire, re-learn.

↺ Maintenance feeds new traces back into Observe — the loop stays open to change and keeps learning.

Two tracks: executable and instructional

Not every crystallization is code. An artifact can save tokens by running with no model, or by being a non-AI instruction — learning captured once as plain text the agent reads and follows, with no AI call to regenerate it. The model still does the task; it just follows a proven recipe instead of re-deriving it.

Executable track

Runs with no model at execution — the work itself is replaced.

cached result · script · hook · MCP tool

Instructional track

A non-AI instruction — captured once, read & followed, no re-synthesis. Two delivery modes:

Recipe / playbook — retrieved just-in-time (skill body, vector-db). ~0 carry until loaded. "to find a specific log, grep logs/ for the request-id…"

Injected instruction — always-on (CLAUDE.md / AGENTS.md / memory), auto-loaded each session. Permanent carry, but cache-friendly at session start.

The skill is the hinge: its body is an instruction, its bundled script is executable. Rule: pick the lowest-carrying-cost option that works — deterministic if mechanical, a retrieved recipe if it needs judgment, an injected instruction only when the learning is worth a permanent per-session tax. Inject sparingly, retrieve by default.

04 · What recurs

Rank candidates by recurrence × cost.

Each operation is normalized to a stable fingerprint (literals → typed placeholders, the SQL-shape / log-template trick), then mined with a frequency threshold — exactly AWO's approach. The top of this ranking is the crystallization queue.

⚠ Illustrative shape. The actual distribution is precisely what Stage 1 measures on the bench corpus.fingerprint = sha1(canonical)[:12]
single calls vs. sequences

A pattern isn't always one call. The richest targets are recurring multi-step sequences — read → edit → run-tests, or "find which file logs X, grep it, extract the line." AWO's state-graph merge collapses equivalent states so the recurring sub-path becomes one artifact. Single calls → rungs 0–2; sequences → skills & tools.

variability decides the rung

Split fixed from varies: varying slots (<PATH>, <STR>, a range) become the artifact's parameters; the fixed structure becomes its body. None → cache/script. A few typed params → MCP tool. Judgment → skill. The mechanical bridge from pattern to typed signature.

06 · Crafting the artifact

Make the skill and the MCP tool best-in-class, not just present.

A generated artifact is only worth its carrying cost if it's authored well. Anthropic's own skill and tool guidance sets the bar the crystallizer must hit.

Rung 3 · the best skill = description + thin how-to + bundled script

Description = retrieval key

Only name + description stay resident. Third person, "what it does AND when," packed with trigger phrases — Claude under-triggers. This is how the next agent knows to use it.

discovery

The LLM writes the how-to

The body carries learned guidance — "to find a specific log, grep logs/ for the request-id, read ±20 lines" — so the next session follows a proven path instead of re-discovering it. Keep it <500 lines; reference files load on demand.

guidance

Push determinism to scripts

Bundle the rung-1/2 script that does the deterministic part — zero tokens until run. Reserve prose for the judgment part only.

cheap path

Rung 4 · the MCP tool, the right way

  • Few, richly-scoped tools — not many micro-tools. One find_x beats list_x+get_x+filter_x — fewer ambiguous decisions, less context.
  • Concise, paginated output with a detail_level knob — Anthropic shows 72 vs 206 tokens for the verbose form (~⅔ off). On-brand for quiet-bash.
  • Strict schemas, self-documenting params (user_id, not user); actionable errors, never raw tracebacks. The fingerprint's varying slots become the input schema.
  • Idempotent & deterministic; annotations for destructive / open-world access; schema-pinned and re-verified on load (defends the "rug-pull").

07 · Trust ladder

A generated artifact is untrusted code until proven.

Promotion is slow and evidence-gated through differential testing against the captured "golden" result. (Caveat from the literature: agent-generated tests are weak validators — use them only as a supplementary signal.)

Proposed

Synthesized, signed & schema-pinned, runs only in a least-privilege sandbox. Never relied upon.

Shadow

Runs in parallel with the LLM loop; outputs diffed against the golden. Pays double — bound the window.

Trusted

Graduates via bounded canary (1–5% → wider). Now replaces the LLM loop; the saving materializes.

Retired

Drift fires (upstream change, TTL lapse, shadow contradiction) → auto-disabled, fall back to LLM, re-queue.

The ladder is deliberately asymmetric: promotion is slow and evidence-gated; demotion is instant and automatic. A drifting artifact fails safe back to the model — never serves stale or poisoned behavior.

08 · Usage feedback & reputation

Every reuse is a vote — even by another agent.

When an artifact is used (including by another session or agent), the use is logged with two signals. They accumulate into a reputation that drives discovery, promotion, and eviction — so good, cheap artifacts rise and drifting ones fall, automatically.

Reputation × description-match is the ranking key. Accumulating good, low-cost uses is exactly the evidence that promotes an artifact up the trust ladder; a run of bad uses or a cost regression demotes and eventually evicts it.

10 · Cost economics

Crystallizing is an investment — only do it past break-even.

A one-time synthesis cost, recovered over many reuses. Stage 2's recurrence × cost ranking is literally estimating this payoff. The decision rule: crystallize when projected_reuses > synthesis_cost / per_use_saving.

tool-maker → tool-user: spend a strong model once, a cheap path uses it forever.LATM: −79% / instance

Three costs the accounting can't ignore

  • Synthesis (one-time) — the strong-model call that writes the artifact. Amortized over reuses.
  • Shadow (transient) — runs alongside the LLM loop, so you pay both. Bound the window or it never nets out.
  • Carrying (ongoing) — every resident tool/skill description costs tokens every turn: ~55K for 58 tools; past a point accuracy also rots.

Why you can't crystallize everything

  • Keep a small resident index (names + descriptions); defer full schemas/bodies to just-in-time search — the vector-db tier.
  • Tool Search loads a ~500-token index vs ~77K — an ~85% cut that also raised accuracy (49%→74%, Opus 4).
  • Net = reuse_saving − synthesis − shadow − carrying. Negative net → evict. Carrying cost makes the active set self-pruning.

11 · Storage

Three layered tiers — files, sqlite, vector-db.

Per-repo by default (same codebase → same answers), with an opt-in, allowlist-only global tier for provably repo-independent facts. Never open, to avoid cross-repo poisoning.

files

The artifacts — executable and instructional — & byte-exact outputs. .quiet-cache/, generated skills/ hooks/ mcp/ recipes/, plus injected CLAUDE.md/AGENTS.md/memory. Zero-dependency, git-reviewable, lossless recovery one Read away.

sqlite

The observability ledger and recurrence ranking. Cheap GROUP BY over fingerprints to answer "what's worth crystallizing." One local dependency.

vector-db

Embeddings of artifact descriptions. Semantic retrieval of which crystallized skill/tool fits a new task — Voyager top-k / CRAFT multi-view — where exact match won't find it.

12 · Roadmap

Measure recurrence before building reuse.

The lowest-risk, highest-information path. Each phase is shippable and earns the next.

1 · Observe-only

Append the fingerprint + cost ledger behind a flag; add it to bench reporting. Output: the recurrence × cost distribution on real sessions — the evidence base for everything else.

risk: loweffort: Szero reuse

2 · Recognition report

A quiet-* helper that prints the top recurring expensive fingerprints per session/repo.

risk: loweffort: Szero reuse

3 · Mechanical crystallization

Narrowest safe slice: input-addressed, per-repo, idempotent-allowlisted ops → cached result or generated script/hook, served at the tail. Wrong-hit rate must be zero.

risk: mediumeffort: Mfirst real saving

4 · LLM-synthesized skills / MCP + vector-db

Separate, opt-in. Each artifact climbs the trust ladder (proposed → shadow → trusted) before it's relied upon.

risk: higheffort: Loff by default

5 · Maintenance loop

Shadow re-checks and drift retirement; keep the archive versioned so artifacts are refined, not just replaced.

risk: mediumeffort: Mkeeps it open

13 · Open questions & risks

What to resolve before any spec.

Open questions

  • Is there enough recurrence to matter? Unknown until Stage 1 runs on the bench corpus. The single most important measurement.
  • PreToolUse skip vs. PostToolUse quiet? Skipping a re-run saves most but may break an adapter's expectations.
  • Who runs LLM synthesis? Async between sessions, or a maintainer command? Decides autonomous vs. supervised.
  • How long is the shadow window? It pays double until promotion — when is the saving net positive?

Risks

  • Under-keying → wrong hits. Over-include inputs, version the schema, deny-list un-enumerable ops.
  • Trusting a bad artifact. Trust ladder + differential gate; generated tests are supplementary only.
  • Security: rug-pull, injection, over-privilege. Sign/pin, sandbox, least privilege, human gate tier-0.
  • Prompt-cache busting. Serve only at the tail; never rewrite old results in v1.
